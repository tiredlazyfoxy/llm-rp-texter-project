# Feature 006 Step 001: Summarization Refinement â€” Backend

## Context

The summarization system is functional but needs refinement: un-summarize support, ability to summarize including the last message (for scene closure), variant-aware compaction, a two-phase LLM flow (memory extraction + summarization), and SSE streaming for the compact endpoint.

## Requirements (backend portion)

1. **Un-summarize** â€” revert the last summary: delete record, unlink messages, return restored messages
2. **Variant-aware compaction** â€” accept `variant_index`, substitute variant content in LLM input
3. **Two-phase LLM flow** â€” Phase 1: memory extraction via `add_memory` tool (OOC/meta observations); Phase 2: narrative summarization with token streaming
4. **SSE streaming compact endpoint** â€” emit phase, tool_call, token, compact_done, error events
5. **Memory extraction prompt** â€” new prompt for Phase 1

## Architecture Decisions

### Compact becomes SSE-streamed
The two-phase flow requires streaming to emit `phase`, `tool_call_start`, `tool_call_result`, and `token` events. Endpoint changes from returning `CompactResponse` JSON to `StreamingResponse` with SSE protocol. Summary text streams token-by-token. Final event: `compact_done` with `CompactResponse` payload.

### Two separate LLM calls
Phase 1 (memory extraction): agentic call with `add_memory` tool via `chat_with_tools`, `stream=False` (only need tool call events). Phase 2 (summarization): streaming `chat()` with `on_delta` callback emitting `token` events. Separate calls keep prompts focused.

### Variant handling
`CompactRequest` gets optional `variant_index`. When provided and the target message is at the current turn, substitute that variant's content in the formatted text sent to the LLM. Source: `generation_variants` JSON on the session.

### Un-summarize is last-only
Backend enforces: only the last summary in the ordered list can be deleted. Uses existing `delete_summary()` which already unlinks messages.

**Details:**

- Route: `DELETE /api/chats/{chat_id}/summaries/{summary_id}` â†’ `list[ChatMessageResponse]`

### 2. Variant-Aware CompactRequest

**Files:**
- `backend/app/models/schemas/chat.py` â€” add `variant_index: int | None = None` to `CompactRequest`
- `backend/app/services/summarization_service.py` â€” accept and use `variant_index`

**Details:**
- In `compact_messages()`, after gathering candidates:
  - If `variant_index is not None` and last candidate is at `chat.current_turn`:
    - Load variants via `_load_variants(chat)` from `chat_service`
    - Get `variants[variant_index].content`
    - Replace last candidate's content in the formatted text (not the DB record â€” only the text sent to LLM)
- Pass `variant_index` from route through to service

### 3. Memory Extraction Prompt

**Files:**
- `backend/app/services/prompts/chat_summarization_prompt.py` â€” add `MEMORY_EXTRACTION_SYSTEM_PROMPT` and `MEMORY_EXTRACTION_USER_PROMPT_TEMPLATE`

**Details:**
```
MEMORY_EXTRACTION_SYSTEM_PROMPT:
- Analyze conversation as a meta-observer
- Identify USER preferences (not character preferences)
- Writing style, pacing, detail level the user enjoys
- Continuation choices revealing what the user finds engaging
- Interaction patterns (how user approaches challenges, NPCs, exploration)
- OOC observations helping build better RP for THIS user
- Do NOT record in-story events or facts â€” those go to the summary
- Use add_memory tool for each observation
- If no observations, that's fine â€” don't force it
```

### 4. Two-Phase SSE Compact Endpoint

**Files:**
- `backend/app/services/summarization_service.py` â€” rewrite `compact_messages` as async generator yielding SSE events
- `backend/app/routes/chat.py` â€” change `compact_chat` to return `StreamingResponse`

**Details:**
The service becomes `compact_messages_stream()` â†’ `AsyncGenerator[str, None]`:

1. Validate session, target message, gather candidates (existing logic)
2. Format messages (with variant substitution from Step 2)
3. **Phase 1 â€” Memory Extraction:**
   - Yield `phase` event: `{"phase": "memory_extraction"}`
   - Build `ToolContext(session_id=session_id)`, call `build_tools(["add_memory"], ctx)` for tool defs + callables
   - Wrap `add_memory` with SSE-emitting wrapper (reuse `_make_tool_wrapper` pattern from `chain_generation_service.py`)
   - The wrapper pushes `tool_call_start` and `tool_call_result` events to the async queue
   - Call `client.chat_with_tools()` with memory extraction prompts, `add_memory` tool, `max_loops=5`, `stream=False`
   - Drain queue â†’ yield tool call SSE events
4. **Phase 2 â€” Summarization (streamed):**
   - Yield `phase` event: `{"phase": "summarization"}`
   - Call `client.chat()` with summarization prompts, `temperature=0.3`, `stream=True`, `on_delta` callback
   - `on_delta` pushes `token` events to the queue â†’ yield as SSE `token` events
   - User sees the summary text being written in real-time
   - Collect all tokens into `summary_content`
5. Create summary record, link messages (existing logic)
6. Yield `compact_done` event with `CompactResponse` payload (includes final complete summary)
7. On error at any point, yield `error` event

**SSE events emitted:**
- `event: phase` / `data: {"phase": "memory_extraction"|"summarization"}`
- `event: tool_call_start` / `data: {"tool_name": "add_memory", "arguments": {...}, "stage_name": "Memory Extraction"}`
- `event: tool_call_result` / `data: {"tool_name": "add_memory", "result": "Memory saved."}`
- `event: token` / `data: {"content": "..."}` â€” summary text streaming during Phase 2
- `event: compact_done` / `data: {"summary": {...}, "updated_message_count": N}`
- `event: error` / `data: {"detail": "..."}`

**Tool wrapper pattern** â€” reuse from `chain_generation_service._make_tool_wrapper` with simplified signature (no `caller_role` filtering; frontend handles debug mode visibility).
