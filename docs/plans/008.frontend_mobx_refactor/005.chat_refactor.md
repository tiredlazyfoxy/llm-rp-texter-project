# Feature 008 Step 005 — Chat Refactor

Goal: explode the singleton `user/stores/ChatStore.ts` (~720 lines) into a per-mount `ChatPageState` owned by `ChatViewPage`. Move every method to external `(state, args, signal)` functions in the same file. Drop chat-list-related state/methods that have moved to `ChatListPageState` in step 004. Delete the singleton + `user/stores/` folder.

This is the largest single refactor in the feature. Behavior parity with the singleton is the bar — no UX changes.

## Why per-mount

- Mount → `loadChat(state, signal)` populates fresh state for the active `chatId`.
- Unmount → cleanup aborts in-flight requests and the active SSE stream cleanly.
- Path-param change (`/chat/123` → `/chat/456`) remounts via router `key={chatId}` set in step 001's wrapper, with no manual reset logic.

## Locked patterns

- `ChatPageState` constructor takes `chatId: string`. State is per-chat and dies on unmount.
- Stream lifecycle: state owns a `streamCtrl: AbortController | null`. `dispose()` method on the class aborts it. Page's `useEffect` cleanup calls `controller.abort()` (for the load) AND `state.dispose()` (for any active stream).
- All API-touching logic is external functions in `chatPageState.ts` — `loadChat`, `sendMessage`, `regenerate`, `regenerateAtTurn`, `compactUpTo`, etc. None on the class.
- Existing `_mergeChatDetail` and `_applyStatUpdate` helpers are kept as **module-private functions** in the same file (preserving the observable-identity optimization for messages/variants/snapshots/summaries — a real perf concern).
- `debugMode` lives on `ChatPageState`, initialized from `localStorage`, persisted on toggle. It's UI-page-scoped, not a global.
- All consuming components receive `state` (or a slice) as an explicit prop. No imports of a singleton anywhere.

## State shape (`chatPageState.ts`)

`ChatPageState` carries the following observable fields, organized by concern:

- **Identity**: `chatId` (constructor arg).
- **Loaded chat data**: `currentChat: ChatDetail | null`, `loadStatus / loadError`, `summaries: ChatSummary[]`, `expandedSummaryMessages: ObservableMap<string, ChatMessage[]>`, `memories: ChatMemoryItem[]`.
- **Send / regenerate streaming**: `isSending`, `pendingInput`, `streamingContent`, `streamingThinking`, `streamingToolCalls`, `isThinking`, `currentPhase`, `currentStatus`.
- **Variants**: `viewingVariantIndex`.
- **Compaction streaming**: `isCompacting`, `compactPhase`, `compactToolCalls`, `compactStreamingContent`, `isRegeneratingSummary`.
- **Generic error**: `error: string | null` (used by retry-after-error).
- **Debug toggle**: `debugMode`.
- **Disposable resource**: non-observable `streamCtrl: AbortController | null`.

Computed (`get`):

- `activeMessages`, `displayItems` (interleaves summaries with messages by turn), `latestTurnVariants`, `hasMultipleVariants`, `currentSnapshot`, `displaySnapshot` (variant-aware), plus convenience getters as needed.

Class method exception: `dispose()` only — aborts `streamCtrl`. Everything else is external.

## External functions (same file)

All take `(state, ...args, signal?)`. AbortSignal is for the load helpers; streaming helpers manage their own controller stored on `state.streamCtrl`.

- `loadChat(state, signal)` — populates `currentChat` + summaries + memories. Uses the merge-in-place helper if `currentChat` already matches `chatId` (re-load case).
- `loadMemories(state, signal)`.
- `sendMessage(state, content, userInstructions?)` — opens SSE stream via `chatApi.sendMessage`, accumulates streaming buffers, captures the previously-viewed variant index for atomic send, clears variants, calls `loadMemories` on done.
- `regenerate(state)` — same shape, current turn.
- `regenerateAtTurn(state, turnNumber)` — same shape, specific turn; reloads detail on done.
- `retryAfterError(state)` — re-sends `pendingInput` if any, else regenerates.
- `stopGeneration(state)` — aborts active stream, resets streaming buffers + flags.
- `continueWithVariant(state, variantIndex, signal)` — POST then reload detail.
- `rewindToTurn(state, turn, signal)` — POST + merge.
- `updateSettings(state, req, signal)` — PUT settings, mirror back to `currentChat.session`.
- `archiveChat(state, signal)` — PUT archive, flip session status.
- `deleteCurrentChat(state, signal)` — DELETE; caller navigates away.
- `editMessage(state, messageId, rawContent)` — PUT then re-send (parses OOC).
- `deleteMessage(state, messageId, signal)` — DELETE + merge.
- Memories: `deleteMemory(state, memoryId, signal)`.
- Summaries: `compactUpTo(state, messageId, variantIndex?)`, `unsummarizeLast(state, summaryId, signal)`, `regenerateSummary(state, summaryId, signal)`, `expandSummary(state, summaryId, signal)`, `collapseSummary(state, summaryId)`.
- Debug: `toggleDebugMode(state)` — flips and persists to localStorage.

Module-private helpers (not exported):

- `mergeChatDetail(state, detail)` — current `_mergeChatDetail` body verbatim; preserves observable identity of nested arrays.
- `applyStatUpdate(state, data)` — current `_applyStatUpdate` body verbatim.

## ChatViewPage rewrite (sketch)

```text
ChatViewPage(props: { chatId: string })
  state = useState(() => new ChatPageState(chatId))
  useEffect mount:
    ctrl = AbortController
    loadChat(state, ctrl.signal)
    cleanup: ctrl.abort(); state.dispose()
  render: layout (header / message history / stats / input) reading state.* via observer
```

Page is the single owner. State is passed down as a prop to chat components. No `chatStore` import anywhere.

## routes.tsx update

Replace the `key`-only `ChatViewPageRoute` from step 001 with:

```text
ChatViewPageRoute = () => {
  const { chatId } = useParams<{ chatId: string }>()
  return <ChatViewPage key={chatId} chatId={chatId!} />
}
```

## Component migrations (consumers of `chatStore`)

After step 003 these live under `user/components/chats/`. Each gets:

- `state: ChatPageState` (or a narrower slice/observable) as an explicit prop.
- `observer` wrapping (some are already wrapped — verify).
- All `chatStore.X` reads replaced with `state.X` (or the slice prop).
- All `chatStore.actionX(...)` calls replaced with `actionX(state, ...)` from `chatPageState`.

Files to update:

- `MessageHistory.tsx` — reads `displayItems`, `streamingContent`, `streamingThinking`, `streamingToolCalls`, `currentPhase`, `currentStatus`, `isSending`, summaries-related fields, `expandedSummaryMessages`, `error`. Calls `expandSummary`, `collapseSummary`, `editMessage`, `deleteMessage`, `regenerateAtTurn`.
- `MessageBubble.tsx` — reads `debugMode`, possibly state-dependent flags. Often takes a single `message` prop too.
- `StatsPanel.tsx` — reads `displaySnapshot`.
- `SummaryBlock.tsx` — reads summary + flags; calls `regenerateSummary`, `unsummarizeLast`, `expandSummary`, `collapseSummary`.
- `ChatMemoriesModal.tsx` — reads `memories`; calls `deleteMemory`.
- `ChatSettingsPanel.tsx` — reads `currentChat.session`, `debugMode`; calls `updateSettings`, `archiveChat`, `toggleDebugMode`.
- `ChatInput.tsx` — already observer; reads `currentChat.session.status`, `isSending`, `pendingInput`, `currentStatus`, `currentPhase`, `activeMessages`. Calls `sendMessage`, `regenerate`, `stopGeneration`. Now receives `state` as a prop instead of importing `chatStore`.

For each, decide between **full state prop** or **slice prop** based on how many fields it reads (architecture's two-pattern guidance). Default to whole-state for tightly-coupled chat children; consider slicing for `MessageBubble` and `StatsPanel` where the surface is narrow.

## Out of scope

- Splitting `chatPageState.ts` into multiple files by concern (load / streaming / summaries). It can stay one large file at first; revisit if it crosses ~1000 lines or becomes painful to read.
- Replacing `console.debug` SSE traces in send/regen handlers — they're useful for debugging streaming and don't violate any rule.
- Field-level error mapping for `editMessage` / `compactUpTo` (current page shows a single error string).
- Optimistic UI changes (e.g., showing the user message instantly on send before the `user_ack`). Current behavior shows after ack; keep parity.
- Refactoring `MessageHistory` into smaller subcomponents — its current shape is fine. If the migration grows it noticeably, split into smaller observer subcomponents in this step; otherwise leave as-is.
