Goal: migrate the remaining three admin pages (`UsersPage`, `LlmServersPage`, `DbManagementPage`) and the three user-management modals to the canonical pattern. Last admin domain.

## Locked patterns

- None of these pages have path params. `routes.tsx` already routes them directly with no wrapper. No changes to `routes.tsx`.
- Each page has a sibling `<page>PageState.ts` exporting class + external `(state, args, signal)` functions.
- All migrated files observer-wrapped; single mount/unmount `useEffect` per page.
- **Modal forms** use **component-local draft classes** (small `useState(() => new XDraft())` per architecture). External `(draft, args, signal)` mutation functions colocated with the modal component.
- `window.confirm`-style action confirmations stay (per current behavior). They're not stateful UI — no class needed.
- Async resource trio everywhere.

## Per-page shape

### `UsersPage` + `usersPageState.ts`

- Fields:
  - `users` trio.
  - `createOpen: boolean`, `passwordTarget: AdminUserResponse | null`, `roleTarget: AdminUserResponse | null` — modal flags / targets.
- External:
  - `loadUsers(state, signal)`.
  - `disableUserAction(state, user, signal)` — confirm before, calls API, refreshes list.

Modal subcomponents take callbacks to refresh the parent list (`onSaved` / `onCreated` → `loadUsers(state, controller.signal)`). Architecture's "no upward callbacks across pages" rule applies to **page→page**; intra-page modal callbacks are fine.

### `LlmServersPage` + `llmServersPageState.ts`

The biggest page in this step — multi-action surface.

- Fields (per current behavior):
  - `servers` trio.
  - **Create draft** (component-local OR page-state — keep it on page state since the create form is inline, not a modal). Or extract to a modal — implementer's call. Default: keep on page state to match current shape.
  - **Edit draft per server**: `editTargetId: string | null`, `editDraft: LlmServerDraft`. Only one server is edited at a time.
  - **Probe results** per server: `probeResults: ObservableMap<string, string[]>` (or an in-memory cache; current behavior re-probes on demand — keep that).
  - **Enabled-models picker state**: `enabledModelsTarget: string | null` (server id whose enabled-models are being edited), `enabledModelsDraft: string[]`.
  - **Embedding model picker**: cluster-wide single setting, plus a draft when editing.
  - `serverErrors`, `saveStatus`, `saveError` for each form scope.
- External:
  - `loadServers(state, signal)`.
  - `createServer(state, signal)`, `updateServer(state, id, signal)`, `deleteServer(state, id, signal)` (with confirm).
  - `probeServer(state, id, signal)` — populates probe-results cache.
  - `saveEnabledModels(state, signal)`, `cancelEnabledModelsEdit(state)`.
  - `setEmbeddingModel(state, id, model, signal)`, `clearEmbeddingModel(state, signal)`.
- Splitting suggestion: extract `LlmServerRow` and a `LlmServerEditForm` as observer subcomponents under `admin/components/llm/` for readability. State stays on the page.

### `DbManagementPage` + `dbManagementPageState.ts`

- Fields:
  - `tables: TableStatus[]` + trio.
  - `actionStatus: AsyncStatus`, `actionError: string | null` — generic "in-flight admin action" trio (covers create-table / sync / reindex / export / import). The current page mostly shows a single error banner; this trio captures it.
  - **Import draft**: `importFile: File | null` (component-local-feel but on page state since the file picker is inline).
- External:
  - `loadDbStatus(state, signal)`.
  - `createTableAction(state, name, signal)`, `syncTableAction(state, name, signal)`, `reindexVectorsAction(state, signal)`.
  - `exportDb(state, signal)` — triggers blob download (the `worlds.ts`/`dbManagement.ts` patterns from step 002).
  - `importDb(state, signal)` — uploads `state.importFile`, then refreshes status on success.

Each action sets `actionStatus = 'loading'` while running, then refreshes table list on success.

## Modal subcomponents — component-local drafts

After step 003 these live at `admin/components/users/`. Each currently uses raw `useState` for form fields. Migrate to the canonical pattern:

### `CreateUserModal`

Sibling state file `createUserDraft.ts` (or inline in the same file — single consumer). Class fields: `username`, `password`, `confirmPassword`, `role`, `serverErrors: Partial<DraftErrors>`. Computed `errors`, `isValid`, `canSubmit`. External `submitCreateUser(draft, signal): Promise<AdminUserResponse | null>`. Modal observer-wrapped; uses `useState(() => new CreateUserDraft())`. Resets on `opened` toggle via `useEffect([opened])` — that single-dep effect is allowed because the draft class lifetime matches the modal's open state, not the component mount.

Actually: simpler — drop the `useEffect([opened])` reset and instead create a fresh draft class on each open via `key={openedToggle}` on the modal contents. Either approach is fine; pick whichever the implementer finds cleaner. Note the choice in `frontend-forms.md` if it deviates.

### `SetPasswordModal`

Same shape: `newPassword`, `confirmPassword`, `serverErrors`. Computed `errors`, `canSubmit`. External `submitSetPassword(draft, userId, signal)`.

### `SetRoleModal`

Same shape: `role: 'admin' | 'editor' | 'player'`, `serverErrors`. Trivial form. External `submitSetRole(draft, userId, signal)`.
