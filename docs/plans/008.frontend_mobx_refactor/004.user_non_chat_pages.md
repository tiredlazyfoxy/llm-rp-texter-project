# Feature 008 Step 004 — User SPA Non-Chat Pages

Goal: migrate the three non-chat user pages to the canonical pattern — page-owned `<Page>State` class, external `(state, args, signal)` mutation functions, single mount/unmount `useEffect`, `observer` wrapping, `useNavigate` for in-SPA links, `useParams` via the route wrapper for path params.

`ChatViewPage` is **not** in this step (chat refactor = step 005).

## Pages migrated

1. `ChatListPage` (`/`) — list "my chats" + delete-with-confirm modal. No path param.
2. `WorldPage` (`/worlds/:worldId`) — single world's name/description + Start New Chat / Edit World buttons.
3. `CharacterSetupPage` (`/worlds/:worldId/new`) — character template form, tool/text model selectors, starting location, submit creates session and navigates.

Each page lives at `user/pages/<Page>.tsx` with a sibling `<page>PageState.ts` that exports the class plus external functions.

## Locked patterns

- Path-param route wrappers in `routes.tsx` extract `useParams()` and pass `worldId` as both `key` and prop. Pages stop reading `window.location.pathname`.
- `WorldPage` and `CharacterSetupPage` accept `worldId: string` prop. `ChatListPage` takes nothing.
- Single `useEffect`, empty deps, mount-load + unmount-abort.
- Async resource trio everywhere: `<name>` / `<name>Status` / `<name>Error`. No `isLoading`.
- Same-SPA navigation via `useNavigate`. Cross-SPA links (`/admin/...`) stay as `<a href>`.
- `CharacterSetupPage` swaps its direct `client.request<{ models }>("/api/chats/models")` (deferred from step 002) for `fetchModelsForSettings()`.

## State shape per page

### `chatListPageState.ts`

- `chats: ChatSessionItem[]` + trio
- `deleteTarget: ChatSessionItem | null`
- `deleteStatus / deleteError`

External:

- `loadChats(state, signal)` — populates the chats trio.
- `deleteSelectedChat(state, signal)` — deletes `state.deleteTarget` via `api/chat.deleteChat`, removes from `chats` on success, resets `deleteTarget`.

### `worldPageState.ts`

- Constructor takes `worldId`.
- `world: WorldInfo | null` + trio.

External:

- `loadWorld(state, signal)` — calls `listPublicWorlds(signal)`, filters by `worldId`. If not found, sets `worldStatus='error'` with "World not found".

(No backend get-by-id endpoint exists; filtering the public list matches current behavior. Adding the endpoint is a backend change out of scope.)

### `characterSetupPageState.ts`

- Constructor takes `worldId`. Initializes `toolModel` / `textModel` from `loadToolModel()` / `loadTextModel()`.
- World data trio (`world`, `worldStatus`, `worldError`).
- Models data trio (`availableModels`, `modelsStatus`, `modelsError`).
- Form draft: `variables: Record<string,string>`, `locationId`, `toolModel`, `textModel`.
- Submit state: `submitStatus`, `submitError`.
- Computeds:
  - `placeholders: string[]` — `{NAME}`-style tokens from `world.character_template`.
  - `canSubmit: boolean` — `world !== null && submitStatus !== 'loading'`.

External:

- `loadCharacterSetup(state, signal)` — fires `loadWorld` and `loadModels` concurrently (`Promise.all`).
- `loadWorld(state, signal)` (internal helper) — populates world trio; on success initializes `variables` from placeholders and `locationId` from first location.
- `loadModels(state, signal)` (internal helper) — populates models trio; if a saved model_id isn't in the available list, falls back to `models[0].model_id` for tool and/or text.
- `submitCharacter(state, signal): Promise<string | null>` — calls `createChat(...)`, returns new session id on success or `null` on error (with `submitError` populated). Page navigates on success.

## Page wiring (sketch, not implementation)

Common shape for all three:

```ts
export const Page = observer(function Page(props) {
  const [state] = useState(() => new PageState(/* props.id */));
  const navigate = useNavigate();
  useEffect(() => {
    const ctrl = new AbortController();
    loadX(state, ctrl.signal);
    return () => ctrl.abort();
  }, []);
  // render based on state.<trio> + handlers that mutate observables / call externals
});
```

Specifics:

- **ChatListPage**: row `onClick` → `navigate('/chat/' + chat.id)`. Trash icon sets `state.deleteTarget`. Modal confirm calls `deleteSelectedChat(state, ctrl.signal)`.
- **WorldPage**: skeleton while loading; error text on error; "Start New Chat" → `navigate('/worlds/'+worldId+'/new')`; "Edit World" remains `<Button component="a" href={'/admin/worlds/'+worldId}>` (cross-SPA, full reload).
- **CharacterSetupPage**: form fields bind to observables (single-field assignment is fine — no `runInAction` needed). Model section onChange both assigns the observable AND calls `saveToolModel/saveTextModel` (localStorage cache stays in sync). Submit handler guards on `state.canSubmit`, then `submitCharacter` and conditional `navigate('/chat/'+id)`.

The `ModelSection` helper used by `CharacterSetupPage` stays inline in the same file (single consumer). Wrap it in `observer` since it reads observable model fields via props.

## routes.tsx update

Replace the `key`-only wrappers from step 001 for `WorldPage` and `CharacterSetupPage` with prop-passing wrappers that read `worldId` via `useParams()` and pass it as `key={worldId}` plus `worldId={worldId!}`.

`ChatViewPageRoute` stays unchanged (refactored in step 005). `ChatListPage` has no wrapper — direct `Route element`.

## Acceptance Criteria

1. Three new state files exist and export the class plus the listed external functions.
2. The three pages are observer-wrapped, use exactly one `useState(() => new ...PageState(...))`, exactly one mount/unmount `useEffect`.
3. `WorldPage` and `CharacterSetupPage` accept `worldId: string` prop; `ChatListPage` accepts no props.
4. `routes.tsx` wrappers extract `worldId` via `useParams` and pass it as both `key` and prop.
5. No `window.location.pathname` reads in any of the three migrated files.
6. Same-SPA navigation uses `useNavigate`. Cross-SPA links remain `<a href>`.
7. `CharacterSetupPage` consumes `fetchModelsForSettings`; no direct `client.request` call.
8. Async resource trios present (no `isLoading: boolean`).
9. `tsc` passes.
10. Smoke-test paths work as described.
11. `user/CLAUDE.md` reflects the new shape.
12. `ChatViewPage`, `ChatStore`, admin SPA, components, etc. — unchanged.
