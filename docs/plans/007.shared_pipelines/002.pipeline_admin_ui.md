Depends on Step 001 (API + types + migration) being merged.

## 8. Decouple AI-assisted pipeline editor from world context

**Why**: pipelines are now shared across worlds, so the LLM-helper that writes pipeline prompts must NOT pull world name / lore / description / NPCs into its system prompt or its tools. A pipeline edited "from world A" must produce identical AI suggestions as the same pipeline edited "from world B" — i.e. the helper must be world-agnostic.

### Current coupling (as found)

The AI helper for pipeline-stage prompts is the same `LlmChatPanel` component used by all admin field editors. It is invoked from `frontend/src/admin/pages/PipelineStageEditPage.tsx` with `fieldType="pipeline_prompt"` and `worldId={worldId}` (line 263–269).

Backend handler: `backend/app/routes/llm_chat.py` (`POST /api/llm/chat`). For any `field_type` (including `"pipeline_prompt"`) it:
1. Loads the world by `req.world_id` (line 87).
2. Loads `world.description`, `world.lore`, all `lore_facts` (injected + non-injected) (lines 92–107).
3. Calls `build_world_field_editor_system(...)` with `world_name`, `world_description`, `world_lore`, `injected_lore` — see `backend/app/services/prompts/world_field_editor_system_prompt.py`. The `pipeline_prompt` role template (lines 48–72) explicitly mentions the world name (`"...for the RPG world \"{world_name}\""`).
4. When `enable_tools=True`, exposes `search`, `get_lore`, `web_search`, etc. via `app.services.admin_tools.get_admin_tools(world_id)` — every research tool is bound to a specific world.

So the pipeline-prompt AI helper currently has FOUR world-specific couplings: world name in role line, world description block, world lore block, world-scoped admin tools.

### What this step changes

**Frontend** — `PipelineStageEditPage.tsx` (after Step 001 rename to live under `/admin/pipelines/:pipelineId/stage/:stageIndex`):
- Stop passing `worldId` to `LlmChatPanel`. Either pass empty string or — preferred — make `worldId` optional in `LlmChatPanel` props and `LlmChatRequest`.
- The "Edit pipeline prompt" page no longer has a world to bind to; it has only a pipeline.

**Frontend** — `frontend/src/admin/components/LlmChatPanel.tsx`:
- Make `worldId` optional in `LlmChatPanelProps`.
- Stop sending `world_id` in the `LlmChatRequest` body when caller didn't supply one (or send an empty string and let the backend decide).

**Frontend** — `frontend/src/types/llmChat.d.ts` / wherever `LlmChatRequest` is typed:
- Make `world_id` optional (`string | null`).

**Backend** — `backend/app/models/schemas/llm_chat.py`:
- Make `world_id: str | None = None`.
- Add `field_type` value `"pipeline_prompt"` to the enum-like comment list (already supported informally).

**Backend** — `backend/app/routes/llm_chat.py`:
- Branch early: `if req.field_type == "pipeline_prompt": ... else: load world ...`. In the pipeline-prompt branch:
  - Do NOT call `worlds_db.get_by_id`.
  - Do NOT load lore facts or world description.
  - Do NOT enable world-scoped admin tools (`get_admin_tools(world_id)`). If `enable_tools` is requested, expose only world-agnostic tools — `web_search` is the only acceptable one. (If the codebase doesn't already split tools that way, add a `get_world_agnostic_tools()` helper in `admin_tools.py` returning only `web_search`.)
  - Build the system prompt via a new dedicated builder: `build_pipeline_prompt_editor_system(current_content, enable_tools)` in `services/prompts/world_field_editor_system_prompt.py` (or split into a new file `pipeline_prompt_editor_system_prompt.py`). It must NOT take `world_name`, `world_description`, `world_lore`, or `injected_lore`.

**Backend** — `backend/app/services/prompts/world_field_editor_system_prompt.py`:
- Remove the `"pipeline_prompt"` entry from `_FIELD_ROLES` and `_FIELD_LABELS` (the existing entry references `{world_name}`).
- The new pipeline-prompt builder (above) replaces the role line with a generic version: e.g. *"You are helping write a pipeline stage prompt. This prompt is the system message for a generation step in an LLM-driven RPG. Pipelines are shared across worlds, so do not assume any specific world setting, lore, or characters."*
- Keep the placeholder list block (`{WORLD_NAME}`, `{RULES}`, `{LOCATION}`, …) — those are runtime injection markers the prompt author writes literally; they are not world content.

**Backend** — `backend/app/services/admin_tools.py`:
- If/when added: `get_world_agnostic_tools()` returning only `web_search` (and the matching subset of `ADMIN_TOOL_DEFINITIONS`). Used by the pipeline-prompt branch in `llm_chat.py`.

### What stays the same

- The world-field editor for `description` / `system_prompt` / `initial_message` (used from `WorldFieldEditPage`) keeps full world context — those fields ARE world-specific.
- `PipelineStageEditPage` keeps its `PlaceholderPanel` and autocomplete — those use the static `PipelineConfigOptions` (placeholders + tool catalog + default templates) which is already world-agnostic.

### Verification

- Open a pipeline stage prompt editor, send the same chat message ("Suggest a planning stage prompt") from two pipelines edited in two different browser tabs without any world context loaded; confirm the system prompt sent to the LLM (visible via backend DEBUG log) does NOT contain world name / description / lore.
- With tools enabled, confirm the only callable tool is `web_search`; `search` / `get_lore` / `get_location_info` / `get_npc_info` / `get_memory` are absent.
- The world field editors (`description`, `system_prompt`, `initial_message`) still receive full world context — regression check.

## Acceptance Criteria

- `/admin/pipelines` shows the list and supports create/edit/delete (delete blocked with friendly message when in use).
- `/admin/pipelines/:id` lets the admin edit name, description, kind, and mode-specific config (system prompt + tools for simple; stage list for chain).
- `/admin/pipelines/:id/stage/:idx` opens the stage prompt editor (same UI as before, sourced from pipeline instead of world).
- World edit page no longer renders generation-mode pickers, system-prompt textarea, simple-tool multi-select, or stage list. It shows a `Pipeline` dropdown and an "Edit pipeline" link.
- Saving a world persists `pipeline_id`; the world's chat respects the picked pipeline (validated against backend Step 001).
- TypeScript build passes (`npx vite build`); no `any`; no references to removed world fields.
- Admin sidebar shows "Pipelines" between "Worlds" and "LLM Servers".
- The AI-assisted pipeline-prompt editor on `/admin/pipelines/:id/stage/:idx` runs without a `world_id`. Its system prompt contains no world name / description / lore / NPC info; with tools enabled, only `web_search` is exposed. World field editors retain full world context.
