# Feature 007 Step 001 — Pipeline Data Model + DB + API + One-Time Migration

Goal: introduce a `pipelines` table, point `World` at it via `pipeline_id`, expose CRUD endpoints, make the runtime read mode/prompt/stages from the pipeline rather than the world, and migrate existing world rows to the new shape via a one-shot manual script.

Pipeline shape (simple / chain / agentic) does not change. Only ownership moves.

```python
from sqlmodel import Field, SQLModel


class PipelineKind(str, enum.Enum):
    simple = "simple"
    chain = "chain"
    agentic = "agentic"


class Pipeline(SQLModel, table=True):
    __tablename__ = "pipelines"

    id: int = Field(primary_key=True)
    name: str = Field(index=True)
    description: str = Field(default="")

    # Simple mode — used when kind == "simple"
    system_prompt: str = Field(default="")
    simple_tools: str = Field(default="[]")          # JSON list of tool names

    # Chain mode — used when kind == "chain"; JSON of PipelineConfig
    pipeline_config: str = Field(default="{}")

    # Agentic mode — used when kind == "agentic"; JSON
    agent_config: str = Field(default="{}")

    created_at: datetime | None = Field(default=None)
    modified_at: datetime | None = Field(default=None)
```

## 8. One-Time Manual Migration

After the schema and code changes above are deployed, existing world rows still have inline pipeline data and `pipeline_id IS NULL`. A one-shot script extracts each world's inline pipeline into a new `pipelines` row and links it via `pipeline_id`. **Manual, paste-runnable, NOT committed to the repo.**

### Pre-conditions

- All code changes from sections 1–7 above are deployed; `init_db()` has run on the target DB (so the `pipelines` table and `worlds.pipeline_id` column exist).
- Application is stopped (no concurrent writes) OR the DB file is copied to a working file first.
- Backup taken:
  ```bash
  cp backend/data/llmrp.db backend/data/llmrp.db.pre007
  ```

### Strategy

For each world row:
1. Read `name`, `system_prompt`, `simple_tools`, `pipeline`, `agent_config`, `generation_mode`.
2. Generate a snowflake ID for the new pipeline (Python `generate_id()`; SQLite has no native snowflake function).
3. Insert one row into `pipelines` with:
   - `name = "<world.name> pipeline"`
   - `description = ""`
   - `kind = world.generation_mode` (or `"simple"` if NULL/empty/invalid)
   - `system_prompt = world.system_prompt`
   - `simple_tools = world.simple_tools`
   - `pipeline_config = world.pipeline`
   - `agent_config = world.agent_config`
   - `created_at = world.created_at`
   - `modified_at = world.modified_at`
4. Update `worlds.pipeline_id = <new id>` for that row.

### Recommended: Python script (paste-runnable)

Cleanest path because snowflake IDs are awkward in raw SQLite. Run from `backend/` with the venv active. Save as `backend/migrate_007.py` (gitignored) or pipe via heredoc — **do NOT commit**.

```python
import asyncio
from datetime import datetime, timezone

from app.db.engine import init_engine, init_db, get_standalone_session
from app.models.pipeline import Pipeline, PipelineKind
from app.models.world import World
from app.services.snowflake import generate_id
from sqlmodel import select


async def main() -> None:
    await init_engine()
    await init_db()
    session = await get_standalone_session()
    async with session:
        all_worlds = list((await session.exec(select(World))).all())
        migrated = 0
        for w in all_worlds:
            if w.pipeline_id is not None:
                continue  # already migrated; idempotent
            kind_str = (w.generation_mode or "simple").strip() or "simple"
            try:
                kind = PipelineKind(kind_str)
            except ValueError:
                kind = PipelineKind.simple
            pipe = Pipeline(
                id=generate_id(),
                name=f"{w.name} pipeline",
                description="",
                kind=kind,
                system_prompt=w.system_prompt or "",
                simple_tools=w.simple_tools or "[]",
                pipeline_config=w.pipeline or "{}",
                agent_config=w.agent_config or "{}",
                created_at=w.created_at or datetime.now(timezone.utc),
                modified_at=w.modified_at or datetime.now(timezone.utc),
            )
            session.add(pipe)
            await session.flush()
            w.pipeline_id = pipe.id
            session.add(w)
            migrated += 1
        await session.commit()
        print(f"Migrated {migrated} worlds.")


asyncio.run(main())
```

The script is idempotent — `if w.pipeline_id is not None: continue` makes a re-run a no-op.

### Sanity-check queries

Open the DB:
```bash
sqlite3 backend/data/llmrp.db
```

Pre-flight (counts; expect ≥ 0):
```sql
SELECT COUNT(*) AS world_count FROM worlds;
SELECT COUNT(*) AS pipeline_count FROM pipelines;
SELECT COUNT(*) AS worlds_without_pipeline FROM worlds WHERE pipeline_id IS NULL;
```

After migration (expect: `pipeline_count == world_count`, `worlds_without_pipeline == 0`):
```sql
SELECT COUNT(*) FROM worlds;
SELECT COUNT(*) FROM pipelines;
SELECT COUNT(*) FROM worlds WHERE pipeline_id IS NULL;

-- Every world resolves to a pipeline of the right kind:
SELECT w.id, w.name, w.generation_mode AS old_mode, p.kind AS new_kind
FROM worlds w LEFT JOIN pipelines p ON p.id = w.pipeline_id;

-- No orphan pipelines (every pipeline points back from at least one world after migration):
SELECT p.id, p.name FROM pipelines p
LEFT JOIN worlds w ON w.pipeline_id = p.id
WHERE w.id IS NULL;

-- Pipeline JSON parses (eyeball a couple of rows):
SELECT id, name, kind, length(system_prompt), length(pipeline_config) FROM pipelines LIMIT 5;
```

### Rollback note

The legacy columns on `worlds` (`system_prompt`, `simple_tools`, `pipeline`, `generation_mode`, `agent_config`) are NOT cleared by this migration — they stay populated as a safety net. Preferred rollback path:

1. Stop the backend.
2. Restore the backup:
   ```bash
   cp backend/data/llmrp.db.pre007 backend/data/llmrp.db
   ```
3. Redeploy the pre-Feature-007 backend.

If the backup is unavailable, partial rollback is possible while the legacy columns still hold the original data:
```sql
UPDATE worlds SET pipeline_id = NULL;
DROP TABLE pipelines;            -- only if nothing else references it
ALTER TABLE worlds DROP COLUMN pipeline_id;  -- SQLite 3.35+
```
This is a last resort; prefer the file-level backup restore.

## Acceptance Criteria

- Fresh DB created via `init_db()` contains a `pipelines` table and `worlds.pipeline_id` column.
- `POST /api/admin/pipelines` creates a pipeline; `PUT` updates it; `DELETE` rejects with 409 if any world references it.
- `World` API responses no longer include `system_prompt`/`simple_tools`/`pipeline`/`generation_mode`/`agent_config`; they include `pipeline_id`.
- A chat against a world with no `pipeline_id` returns 400 with "World has no pipeline configured".
- A chat against a world referencing a `kind="simple"` pipeline runs the simple generation flow using `pipeline.system_prompt`/`pipeline.simple_tools`.
- A chat against a world referencing a `kind="chain"` pipeline runs the chain flow using `pipeline.pipeline_config`.
- DB export contains `pipelines.jsonl.gz`; round-trip import recreates pipelines and reattaches worlds to them via `pipeline_id`.
- Frontend builds with strict TypeScript (no `any`); world API types compile against new shape.
- After running the migration script: `SELECT COUNT(*) FROM worlds WHERE pipeline_id IS NULL` returns 0; `pipelines` row count equals world count (one pipeline per world; sharing comes later via UI).
- Every `pipelines.kind` value matches the original `worlds.generation_mode` (or `"simple"` when blank).
- `pipelines.system_prompt`, `simple_tools`, `pipeline_config`, `agent_config` are byte-identical to the originating world's columns.
- A second run of the migration script is a no-op (`Migrated 0 worlds.`).
- Chat works against migrated worlds without further code changes.
