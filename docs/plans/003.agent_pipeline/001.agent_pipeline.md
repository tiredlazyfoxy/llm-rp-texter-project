# Feature 003 Step 001 -- Two-Stage Agent Pipeline (Thinking + Writing)

## Context

The current chat system (feature 002) uses a single LLM call with tools to generate responses. This conflates decision-making (tool calls, stat analysis, planning) with prose writing. Feature 003 splits the agent into a two-stage pipeline:

1. **Thinking Agent** -- Has tools, gathers context iteratively, makes decisions, outputs structured JSON plan + stat updates.
2. **Prose Writer** -- Receives chat history + the plan, writes narrative prose. No tools. Different model.

This enables using a fast/cheap reasoning model for thinking and a creative model for writing.

### Dependencies

- Feature 002 Steps 001-004 (chat DB, tools, API+UI, summarization) fully implemented.
- `World.pipeline` field already exists as opaque JSON string (feature 001 step 002).

## 1. Pipeline Configuration

### 1a. World.pipeline JSON Schema

The `World.pipeline` field gets a defined schema. Empty `{}` = single-step legacy mode.

```
PipelineConfig:
  enabled: bool (default false)
  thinking:
    llm_model_id: str | null      # null = use session default
    temperature: float | null      # null = use session default
    max_tool_loops: int (default 10)
  writing:
    llm_model_id: str | null
    temperature: float | null
```

Validated via Pydantic `PipelineConfig(BaseModel)` in `backend/app/models/schemas/pipeline.py`.

### 1b. ChatSession Overrides

New columns on `ChatSession`:

| Column | Type | Notes |
|---|---|---|
| `thinking_model_id` | str or null | Override thinking model. Null = use world default. Editor+ only. |
| `thinking_temperature` | float or null | Override thinking temperature. Null = use world default. |
| `writing_model_id` | str or null | Override writing model. Null = use world default. Editor+ only. |
| `writing_temperature` | float or null | Override writing temperature. Null = use world default. |
| `pipeline_enabled` | bool or null | Override pipeline enabled. Null = use world default. Editor+ only. |

**Access control**:
- **Model selection overrides**: Editor+ role only. Regular users use world-configured models.
- **Temperature overrides**: Allowed for all users, separate per stage.
- **Pipeline enabled toggle**: Editor+ only.

### 1c. Resolution

```
resolve_pipeline(world, session) -> ResolvedPipeline
```

World pipeline is the base. Session columns (when not null) override corresponding fields. If resolved `enabled` is false, falls back to existing single-step flow.

## 2. Thinking Agent (Stage 1 -- Decision Making)

### 2a. Purpose

Receives full context + tools. Calls tools iteratively (multi-loop, configurable `max_tool_loops`). Outputs structured JSON generation plan.

### 2b. Prompt

**File**: `backend/app/services/prompts/thinking_agent_system_prompt.py`

New constant `THINKING_AGENT_SYSTEM_PROMPT` with full documentation header (PURPOSE, USAGE, VARIABLES, DESIGN RATIONALE, CHANGELOG — see `docs/plans/backlog/prompt_tuning.md`). Same variable interpolation as current `CHAT_SYSTEM_PROMPT` (world, location, NPCs, rules, character, stats). Different instructions:

- "You are the PLANNING agent. Analyze the player's action, gather information using tools, produce a structured generation plan."
- "Do NOT write narrative prose. Only produce the JSON plan."
- Instructions for what the JSON output must contain.

### 2c. Structured JSON Output

The thinking agent uses `chat_json()` (or `chat_with_tools()` with JSON mode) to produce structured output. Output schema:

```
GenerationPlanOutput:
  collected_data: str         # Relevant text from tools/context the writer needs
  stat_updates: list[{name: str, value: str}]  # Stats that changed. Empty if none.
  decisions: list[str]        # Bullet points: what happens this turn
```

Pydantic model used as JSON schema for the LLM call. Parsed and validated after thinking completes.

**Fallback**: If JSON parsing fails, log warning and attempt regex extraction. If that fails too, yield error SSE event.

### 2d. Tool Loop

Uses `chat_with_tools()` with `max_loops` from pipeline config (default 10). Agent can:

1. Call `get_location_info`, `get_npc_info`, `search` iteratively
2. Build up context across multiple tool calls
3. Final response is the JSON generation plan

### 2e. Stat Updates

Stats are extracted from `stat_updates` in the JSON output (not from `[STAT_UPDATE]` blocks). Validated and applied using existing `validate_and_apply_stat_updates()` after thinking completes, before writing starts.

## 3. Prose Writer (Stage 2 -- Text Generation)

### 3a. Purpose

Receives clean chat history + generation plan. Writes **pure narrative prose text** -- no tools, no structured blocks, no stat updates, no metadata. The writer's entire output becomes the visible chat message content.

Stats are NOT in the writer's output. They come from the thinking agent's JSON plan and are sent to the client as a separate `stat_update` SSE event before the writing phase begins.

### 3b. Prompt

**File**: `backend/app/services/prompts/prose_writer_system_prompt.py`

New constant `PROSE_WRITER_SYSTEM_PROMPT` with full documentation header (PURPOSE, USAGE, VARIABLES, DESIGN RATIONALE, CHANGELOG):

- World description + character info (lighter context than thinking agent -- no stat definitions, no tool instructions)
- "Follow the generation plan faithfully. Do not add/remove/change plot points."
- "Write immersive prose. Include all NPC dialogue specified in the plan."
- "Your output is ONLY narrative prose. Do NOT include stat updates, JSON, tags, tool calls, or meta-information."

**File**: `backend/app/services/prompts/prose_writer_plan_message.py`

New constant `PROSE_WRITER_PLAN_MESSAGE` with full documentation header -- template for injecting plan into writer context:

- Collected context section
- Decisions section

### 3c. Writer Input Construction

`build_writer_context(session, messages, plan, world, db)`:

1. System message: `PROSE_WRITER_SYSTEM_PROMPT` with world + character vars
2. Chat history: Only user + assistant messages (active variants, not summarized). Summaries as system messages. NO tool calls, NO thinking content -- clean prose only.
3. Latest user message
4. Plan message: System message with `PROSE_WRITER_PLAN_MESSAGE` containing collected_data + decisions

### 3d. Writer Call

Simple streaming LLM call, no tools:

```
client.chat(model=writing_model_id, messages=writer_messages, temperature=writing_temperature, stream=True)
```

## 4. SSE Streaming Protocol

### 4a. New Events

| Event | Data | When |
|---|---|---|
| `phase` | `{"phase": "thinking" \| "writing"}` | Pipeline phase transition |
| `status` | `{"text": "..."}` | Human-readable status for regular users (e.g. "Searching world lore...", "Planning response...", "Writing...") |

### 4b. Event Sequence (Pipeline Mode)

```
phase("thinking") -> status("Gathering context...") ->
  [tool_call_start -> tool_call_result -> status("Searching...")]* ->
  status("Planning response...") -> thinking* ->
  stat_update ->
phase("writing") -> status("Writing...") ->
  token* ->
done
```

**Stats flow**: `stat_update` is always emitted between thinking and writing phases (even if no stats changed -- sends current stats). The client updates the stats panel immediately when this event arrives, before prose starts streaming. This means the user sees updated stats while the prose is being written.

### 4c. Event Sequence (Legacy Single-Step)

Unchanged: `thinking* -> thinking_done -> tool_call_start -> tool_call_result -> token* -> stat_update? -> done`

### 4d. Visibility Rules

- **Regular users**: See `status` events only (rendered as status indicator text). Tool calls, thinking tokens, and generation plan are NOT streamed to them.
- **Editor+ users with debug toggle on**: See all events -- tool_call_start/result, thinking tokens, generation_plan. Shown as collapsible sections.
- The `phase` event is always sent (used by frontend for UI state transitions regardless of role).

## 5. Stat Visibility (Cross-Cutting)

This is a small addition that applies to all stages (single-step and pipeline).

### 5a. WorldStatDefinition -- New Column

| Column | Type | Default | Notes |
|---|---|---|---|
| `hidden` | BOOLEAN | false | If true, stat is hidden from regular users but still used in rules and planning |

- **Hidden stats**: Not shown in the user-facing StatsPanel. Still included in the LLM system prompt (both thinking agent and single-step agent need all stats for decision-making). Still included in `stat_update` SSE events (but frontend filters them out for display).
- **Visible stats** (`hidden=false`, the default): Shown to users as before.
- **Debug mode** (editor+ with debug toggle): Shows ALL stats including hidden ones in the StatsPanel.

### 5b. Affected Components

| Component | Change |
|---|---|
| `WorldStatDefinition` model | Add `hidden: bool = False` column |
| `StatDefinitionResponse` schema | Add `hidden: bool` field |
| `StatDefinition` TS interface | Add `hidden: boolean` field |
| `StatsPanel` component | Filter out hidden stats unless debug mode is on |
| `stat_update` SSE event | Sends all stats (hidden and visible). Frontend filters display. |
| `format_stat_definitions()` helper | Include hidden flag in LLM context so the agent knows which stats are hidden |
| Admin world editor | Checkbox on each stat definition: "Hidden from players" |
| DB import/export | Include `hidden` field in WorldStatDefinition export/import |

### 5c. Why Frontend Filtering (Not Backend)

The `stat_update` event sends ALL stats because:
1. Editors in debug mode need to see hidden stats
2. Hidden stats may still be needed for future client-side logic (e.g. conditional UI)
3. Simpler backend -- one code path for all roles

The StatsPanel checks each stat against the stat definitions and the user's role + debug toggle to decide what to show.

## 6. Service Layer

### 6a. New File: `backend/app/services/pipeline_service.py`

```
generate_pipeline_response(session_id, user_message, db) -> AsyncGenerator[SSEEvent]
```

Flow:

1. Load session, world, resolve pipeline config
2. Save user message
3. **Thinking phase**:
   - Yield `phase("thinking")`
   - Build context via existing `build_chat_context()`
   - Get tools via existing `get_chat_tools()`
   - Get thinking LLM client
   - Call LLM with tools + JSON output schema, `max_loops` from config
   - Yield `status` events for tool calls (for regular users)
   - Yield `tool_call_start/result` and `thinking` events (for debug mode)
   - Parse JSON generation plan from response
   - Validate + apply stat updates
   - Yield `stat_update`
4. **Writing phase**:
   - Yield `phase("writing")`, `status("Writing...")`
   - Build writer context via `build_writer_context()`
   - Get writing LLM client
   - Stream prose, yield `token` events
5. **Finalize**:
   - Save assistant ChatMessage with `content` = prose, `tool_calls` = from thinking, `generation_plan` = JSON plan
   - Increment turn, save snapshot
   - Yield `done`

### 6b. Modify: `backend/app/services/chat_service.py`

`generate_response()` dispatches based on resolved pipeline config:

- `pipeline.enabled` = true -> `generate_pipeline_response()`
- `pipeline.enabled` = false -> existing single-step flow (extracted into `generate_single_step_response()`)

Regeneration: Same dispatch logic. Both stages re-run on regenerate.
Rewind: No changes needed -- works identically.

## 10. Error Handling

- **Thinking JSON parse failure**: Log warning, attempt fallback extraction, yield `error` SSE event if unrecoverable
- **Writer failure**: Yield `error` event, don't save message. Stats from thinking already applied -- on retry (regenerate), stats revert to pre-turn snapshot first (existing regen logic)
- **Tool errors during thinking**: Individual tool errors returned to agent as error results (existing behavior), agent continues
