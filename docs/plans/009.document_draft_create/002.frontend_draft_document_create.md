# Step 002 — Frontend: draft document create flow

## Goal

Replace the "create row immediately, then navigate" behavior in `WorldViewPage`
with a draft flow: pre-allocate a snowflake id, navigate to the editor with
`?new=1`, and persist only on Save (with queued link ops flushed afterwards).

## Files to create or modify

- `frontend/src/api/admin.ts` — add `getNewSnowflakeId()`. (If existing
  precedent prefers a sibling module, e.g. there is already an `ids.ts`-like
  pattern, place it there. Coder picks based on existing layout — strong
  preference for the lowest-friction option.)
- `frontend/src/types/world.d.ts` — add optional `id?: string` to
  `CreateDocumentRequest` (line 62-90 area).
- `frontend/src/admin/pages/WorldViewPage.tsx` — rewrite `handleCreate`
  (~line 277-285): call `getNewSnowflakeId()`, then navigate to
  `/admin/worlds/{worldId}/documents/{newId}/edit?new=1&doc_type={type}`.
  No backend create call here.
- `frontend/src/admin/pages/worldViewPageState.ts` — **delete**
  `createNewDocument` (line 223-248). Dead code is forbidden per project
  conventions.
- `frontend/src/admin/routes.tsx` — `DocumentEditPage` route wrapper reads
  `?new=1` (boolean) and `?doc_type=...` from `useSearchParams`, passes
  `isNew` and `initialDocType` props (and continues to provide the existing
  `key={id}` remount semantics).
- `frontend/src/admin/pages/DocumentEditPage.tsx` — accept and forward the new
  props to the page state. Preserve the existing layout and bindings.
- `frontend/src/admin/pages/documentEditPageState.ts` — add draft mode:
  - Constructor accepts optional `isNew: boolean` and `initialDocType?: string`.
  - When `isNew`: skip `loadDocument`; initialize an empty draft of
    `initialDocType`; set `loadStatus = 'ready'`; expose `isNew` as an
    observable.
  - Add `pendingLinkOps` observable (queue of link create/delete intents).
  - Existing `createLink` / `deleteLink` paths (line 196-198 area): when
    `isNew`, mutate the queue and the locally-displayed link list instead of
    calling the backend.
  - `saveDocument`: when `isNew`, POST `worldsApi.createDocument(worldId,
    { id: docId, doc_type, name, content, exits? }, signal)`; on success
    replay queued link ops via the existing link APIs (sequential is fine,
    abort-aware via `signal`); clear `pendingLinkOps`; set `isNew = false`.
- `frontend/src/admin/pages/CLAUDE.md` — add a one-line note about the draft
  flow and `?new=1` flag if the file lists patterns; otherwise leave it.

## Signatures

API client:

```
// frontend/src/api/admin.ts (or sibling ids.ts)
export async function getNewSnowflakeId(signal?: AbortSignal): Promise<string>
```

Returns the `id` string from `GET /api/admin/snowflake/new`.

Type change:

```
// frontend/src/types/world.d.ts
export interface CreateDocumentRequest {
  id?: string;          // NEW — optional client-supplied snowflake id
  doc_type: ...;
  name: ...;
  content: ...;
  exits?: ...;
}
```

State change (sketch — no implementation):

```
class DocumentEditPageState {
  constructor(
    worldId: string,
    docId: string,
    options?: { isNew?: boolean; initialDocType?: DocType },
  );
  isNew: boolean;                 // observable
  pendingLinkOps: LinkOp[];       // observable
  // existing fields unchanged
}
```

`LinkOp` is a small discriminated union representing queued
create/delete intents (shape decided by the coder; must capture enough to
replay via the existing link APIs).

`WorldViewPage.handleCreate` becomes:

```
const id = await getNewSnowflakeId();
navigate(`/admin/worlds/${worldId}/documents/${id}/edit?new=1&doc_type=${type}`);
```

## Tests

No new automated tests (per feature 008 convention). The step file must
include a manual smoke-test checklist in the PR / status.md notes:

1. Click "New Location" — URL becomes
   `/admin/worlds/<wid>/documents/<id>/edit?new=1&doc_type=location` and the
   page shows an empty draft.
2. Type a name + content, then Save — row is created in the DB with that exact
   id. URL stays the same. `?new=1` becomes irrelevant after save (state's
   `isNew` flips to `false`).
3. Repeat with NPC and create a link to a location while the NPC is still
   a draft — the link displays in the editor but the network panel shows no
   link API calls. On Save, the document POST happens first, then the link
   POSTs flush.
4. Click "New Location", reload the page on `?new=1` URL — page shows a fresh
   empty draft with the same id and doc_type. Draft content is gone (expected).
5. Cancel/close the tab without saving — no orphaned row in the DB.

## Definition of done

- `WorldViewPage.handleCreate` performs no document POST; it only fetches an
  id and navigates.
- `worldViewPageState.createNewDocument` is removed; no callers remain.
- `DocumentEditPage` route wrapper passes `isNew` and `initialDocType` from
  query params.
- `DocumentEditPageState` honors `isNew`: skips load, starts an empty draft,
  queues link ops, and on Save persists the document with the pre-allocated
  id and replays queued link ops.
- After a successful Save, `isNew` is `false` and subsequent edits use the
  existing update path.
- Reload on `?new=1` re-initializes a fresh empty draft (same id and
  doc_type) without errors.
- No `any` introduced; the new `id?: string` field is reflected in the
  TypeScript type.
- No leftover dead code (`createNewDocument` deleted, not commented out).
- Smoke-test checklist above passes.

## Dependencies

Step 001 (backend endpoint and `id`-accepting `CreateDocumentRequest`) must be
merged first; the frontend cannot work without both.
