# Feature 001 Step 001 — Login, User Model, DB Bootstrap

## Context

First implementation step. The project has boilerplate (FastAPI structure, Vite multi-page setup) but no models, routes, or services. This step establishes: user model, ID generation, database lifecycle, authentication API, and the login/setup UI.

---

## 1. Snowflake ID Generator

**File**: `backend/app/services/snowflake.py`

Bit layout (63 bits, signed int64):
- 41 bits: milliseconds since epoch (2025-01-01) — ~69 years
- 19 bits: sequence counter (0–524287 per ms)
- 3 bits: machine ID (fixed 0, supports up to 8 instances)

- Singleton generator, async-safe
- `generate_id() -> int`
- Stored as INTEGER in SQLite, serialized as string in all API responses (JS can't handle int64)

## 2. User Model

### SQLModel (backend) — `backend/app/models/user.py`

| Column | Type | Notes |
|---|---|---|
| `id` | int (int64 snowflake) | Primary key, stored as INTEGER |
| `username` | str | Unique, indexed |
| `pwdhash` | str \| None | None = disabled user |
| `salt` | str \| None | App-level salt, prepended before bcrypt; None = disabled |
| `role` | UserRole enum | admin, editor, player (no role) |
| `jwt_signing_key` | str \| None | Per-user HS256 secret |
| `last_login` | datetime \| None | Updated on each successful login |
| `last_key_update` | datetime \| None | When jwt_signing_key was last generated |

**UserRole enum**: admin > editor > player (inherited top-down; admin has editor+player access, editor has player access)

**Disabled user**: pwdhash=None, salt=None → cannot log in

### Pydantic API schemas — same file or `backend/app/models/schemas/auth.py`

- `UserResponse(BaseModel)`: id (str), username, role
- `LoginRequest(BaseModel)`: username, password
- `LoginResponse(BaseModel)`: token (str)
- `CreateDBRequest(BaseModel)`: admin_username, password, password_confirm
- `AuthStatusResponse(BaseModel)`: needs_setup (bool)

### TypeScript — `frontend/src/types/user.d.ts`

- `id`: string (int64 as string)
- `username`: string
- `role`: "admin" | "editor" | "player"
- No secrets exposed to frontend

## 3. Password Hashing

**Dependency**: `passlib[bcrypt]` (add to pyproject.toml)

- App-level salt (random, stored in user.salt) prepended to password before bcrypt
- `hash_password(password, salt) -> str`
- `verify_password(password, salt, hash) -> bool`
- On user create: generate random salt, hash password
- On password change: generate new salt, rehash

---

## 4. JWT Integration

**File**: `backend/app/services/auth.py`

- **Algorithm**: HS256 with per-user signing key (no global secret)
- **Token payload**: user_id (str), username, role, exp
- `create_token(user) -> str`
- `verify_token(token) -> payload` — decode without verification first to get user_id, load user from DB, then verify with user's `jwt_signing_key`
- **Key rotation**: on login, if `last_key_update` > 30 days old → regenerate key, update timestamp
- **Password change**: always regenerate `jwt_signing_key` (invalidates all sessions)
- FastAPI dependency: `get_current_user()` for protected routes (used in later steps)

---

## 5. Database Service

**File**: `backend/app/services/database.py`

- SQLModel async engine (aiosqlite)
- `has_database() -> bool` — checks DB file exists with tables
- `create_database(admin_username, password)` — create all tables + admin user
- `import_database(zip_file)` — create tables + import from zip
- App startup: check DB, expose status for `/api/auth/status`

---

## 6. DB Import/Export Service

**File**: `backend/app/services/db_import_export.py`

- Format: single `.zip` containing one `.jsonl.gz` per table
- One function per table: `export_users()`, `import_users()`
- Top-level: `export_all() -> zip bytes`, `import_all(zip_file)`
- Extended as new models are added in future steps

## 7. Login API

**File**: `backend/app/routes/auth.py`

### Endpoints

| Method | Path | Purpose |
|---|---|---|
| GET | `/api/auth/status` | Returns `{ needs_setup: bool }` |
| POST | `/api/auth/login` | Standard login → JWT |
| POST | `/api/auth/setup/create` | Create DB + admin user → JWT |
| POST | `/api/auth/setup/import` | Import DB from zip upload |

### Login security

- **Rate limit**: max 4 sequential wrong attempts per username → 1-min cooldown (in-memory dict)
- During cooldown: same "wrong credentials" response (indistinguishable from real failure)
- **Timing**: random 100–300ms `asyncio.sleep()` before every response
- Setup endpoints: only available when `needs_setup=true`, reject otherwise

## 8. Login Page (Frontend)

**Location**: `frontend/src/login/` — separate entry point, NOT part of User or Admin SPA

### New files
- `frontend/login/index.html` — HTML entry
- `frontend/src/login/main.tsx` — React mount
- `frontend/src/login/Login.tsx` — Main component

### Update
- `frontend/vite.config.ts` — add login as third page in multi-page build

### Behavior
1. On load: `GET /api/auth/status`
2. If `needs_setup=true` → setup UI with two tabs:
   - **Create DB**: admin username + password + confirm password → `POST /api/auth/setup/create`
   - **Import DB**: file upload (zip) → `POST /api/auth/setup/import`
3. If `needs_setup=false` → login form: username + password → `POST /api/auth/login`
4. On success: store JWT in localStorage, redirect to `/` (user SPA)
5. UI: Mantine components
