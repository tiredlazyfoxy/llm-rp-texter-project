# Feature 004 Step 004: Tool-Based Planning Context

## Context

The chain generation pipeline's planning stage currently asks the LLM to output a `GenerationPlanOutput` JSON blob after calling tools. This is unreliable — LLMs wrap JSON in markdown fences, truncate output, add commentary, or produce malformed JSON. The `_parse_plan_json()` function has multiple fallback strategies but still fails regularly.

**New approach**: Instead of asking the LLM to produce JSON, give it **planning tools** that build the structured output. The LLM calls `add_fact()`, `add_decision()`, `update_stat()` — the same way it already calls `get_location_info()` or `move_to_location()`. The LLM's final text response is ignored. All structured data flows through tool calls.

## Design

### PlanningContext Object

```python
class PlanningContext(BaseModel):
    facts: list[str] = []
    decisions: list[str] = []
    stat_updates: list[StatUpdateEntry] = []
```

Created before the planning LLM call. Mutated by planning tools. Converted to `GenerationPlanOutput` after planning completes (for persistence on `ChatMessage.generation_plan`):
- `collected_data` = `"\n".join(context.facts)`
- `decisions` = `context.decisions`
- `stat_updates` = `context.stat_updates`

### Planning Tools (3 new tools)

| Tool | Params | Behavior |
|------|--------|----------|
| `add_fact(content)` | `content: str` | Appends to `planning_context.facts`. Returns confirmation. |
| `add_decision(content)` | `content: str` | Appends to `planning_context.decisions`. Returns confirmation. |
| `update_stat(name, value)` | `name: str, value: str` | Validates immediately via `validate_and_apply_stat_updates()`. On success: appends to `planning_context.stat_updates`, mutates char/world stats dicts. Returns "Stat updated: {name} = {value}". On failure: returns error message to LLM (LLM can self-correct). |

These are **planning-only** — not added to the standard 8 chat tools. A new factory `get_planning_tools()` returns all 8 chat tools + 3 planning tools.

### Pipeline Flow (Before → After)

**Before:**
```
Planning LLM + read tools → raw text with JSON → _parse_plan_json() → GenerationPlanOutput
  → validate stats → build_writing_plan_message() → Writer LLM
```

**After:**
```
Planning LLM + read tools + planning tools → PlanningContext (built by tool calls)
  → convert to GenerationPlanOutput → build_writing_plan_message() → Writer LLM
```

### Future: Splittable Stages

The tool-based approach makes the pipeline trivially splittable:

```
Stage: "research"  (tools: read + add_fact)       → gather context
Stage: "planning"  (tools: add_decision + update_stat) → make decisions
Stage: "writing"   (tools: read-only)              → write prose
```

Not implemented in this step — but the architecture supports it. For now, `"planning"` does all three (research + decisions + stats) in one stage, same as today.

## Changes

### 1. `backend/app/models/schemas/pipeline.py` — Add PlanningContext

- Add `PlanningContext(BaseModel)` with `facts`, `decisions`, `stat_updates` fields
- `GenerationPlanOutput` stays as-is (used for persistence)
- `PipelineStage`, `PipelineConfig` unchanged

### 2. `backend/app/services/chat_tools.py` — Planning tool definitions + factory

Add 3 Pydantic param schemas: `AddFactParams`, `AddDecisionParams`, `UpdateStatParams`

Add 3 tool definitions via `pydantic_to_openai_tool()`:
- `add_fact` — "Record a relevant observation or context the writing agent needs"
- `add_decision` — "Record a specific plot point or action outcome for this turn"
- `update_stat` — "Update a stat value (validated immediately)"

Add new factory:
```python
def get_planning_tools(
    world_id: int,
    session_id: int,
    planning_context: PlanningContext,
    stat_defs: list[WorldStatDefinition],
    char_stats: dict,
    world_stats: dict,
) -> tuple[list[dict], dict[str, Callable]]:
```

Returns 8 existing chat tools + 3 planning tools. Planning tool callables are closures that mutate the `PlanningContext` and stats dicts.

### 3. `backend/app/services/prompts/planning_system_prompt.py` — Replace JSON instructions

**Remove**: The entire "Output Format" section (JSON schema, field descriptions, "Output ONLY the JSON object" instruction).

**Replace with** "Building Your Plan" section:
1. Research: call read tools to gather info
2. Record context: call `add_fact(content)` for each piece the writer needs
3. Make decisions: call `add_decision(content)` for each plot point
4. Update stats: call `update_stat(name, value)` — validated immediately
5. Save memories: call `add_memory(content)` as before

Emphasize: "You MUST use add_fact and add_decision tools. Do NOT output JSON. Your text response is ignored — only tool calls matter."

Update role description: "produce a structured JSON plan" → "use planning tools to build context for the writing agent"

Update Available Tools section to include the 3 new tools.

### 4. `backend/app/services/chain_generation_service.py` — Core pipeline changes

**Planning stage changes:**

- Create `PlanningContext()` before LLM call
- Load `char_stats`/`world_stats` dicts early (moved up)
- Use `get_planning_tools()` instead of `get_chat_tools()`
- **Remove** `_parse_plan_json()` function entirely
- **Remove** JSON parse error path ("Planning stage produced invalid JSON")
- After LLM call: convert `PlanningContext` → `GenerationPlanOutput` for persistence
- **Remove** post-planning `validate_and_apply_stat_updates()` call (stats already validated by `update_stat` tool)
- `new_char`/`new_world` are the mutated stats dicts directly

**Writer stage**: Unchanged — `build_writing_plan_message()` still receives `collected_data` and `decisions`, just sourced from tool calls instead of parsed JSON.

### 5. SSE / Frontend — No changes needed

Planning tools go through existing `_make_tool_wrapper`:
- Editor+: sees `tool_call_start`/`tool_call_result` for each planning tool
- Players: sees status messages

`ChatMessage.generation_plan` still stores `GenerationPlanOutput` JSON — same schema. Frontend `GenerationPlanData` interface unchanged. Debug mode shows plan data as before.

`PipelineConfig` schema unchanged — admin pipeline editor works as-is.

## Verification

1. **Chain mode chat**: Send a message, verify server logs show `add_fact`, `add_decision`, `update_stat` tool calls
2. **Writer receives plan**: Verify writing stage gets facts/decisions via `build_writing_plan_message()`
3. **Stats validated at call time**: Send message that triggers stat change, verify stat is validated immediately (check debug logs)
4. **Debug UI**: Check that completed message's generation_plan in debug mode shows collected_data (joined facts) and decisions
5. **Existing pipeline configs**: Verify existing world with chain mode + planning/writing stages works without migration
6. **Edge case — LLM doesn't call tools**: PlanningContext empty → writer gets empty plan → still produces output (degraded but not error)
7. **Edge case — stat validation failure**: LLM calls `update_stat("nonexistent", "5")` → gets error message → can retry or skip

## Risks

- **LLM may not call planning tools**: Prompt must strongly instruct. Worst case = empty plan, writer produces generic output (no worse than current JSON parse failure).
- **More tool roundtrips**: Planning tools are local operations (list append, dict update) — negligible overhead vs existing vector search tools.
- **Quality depends on LLM tool use**: In practice, tool calling is more reliable than structured JSON output for most models.
