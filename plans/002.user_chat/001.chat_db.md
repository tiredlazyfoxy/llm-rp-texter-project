# Feature 002 Step 001 — Chat Database Models

## Context

Feature 002 implements the user-facing chat flow. This first step defines all database models needed for the chat system: sessions, messages, state snapshots, and summaries. Models and JSONL import/export only — no API routes or UI.

---

## 1. Chat Session Model

Represents an ongoing RP chat between a user and a world. Created when a user selects a world, fills in character template variables, picks a starting location, and begins.

**File**: `backend/app/models/chat_session.py`

### Table: `chat_sessions`

| Field | Type | SQLite Type | Notes |
|---|---|---|---|
| `id` | int64 (snowflake) | INTEGER | Primary key |
| `user_id` | int64 | INTEGER | FK → users.id, indexed |
| `world_id` | int64 | INTEGER | FK → worlds.id, indexed |
| `current_location_id` | int64 \| None | INTEGER | FK → world_locations.id, nullable |
| `character_name` | str | TEXT | Filled from `{NAME}` placeholder |
| `character_description` | str | TEXT | Fully resolved character_template with all placeholders filled |
| `character_stats` | str | TEXT | JSON object `{"stat_name": value, ...}` |
| `world_stats` | str | TEXT | JSON object `{"stat_name": value, ...}` |
| `current_turn` | int | INTEGER | Starts at 0, incremented after each user+assistant exchange |
| `status` | str | TEXT | `"active"` / `"archived"` |
| `llm_server_id` | int64 \| None | INTEGER | FK → llm_servers.id |
| `llm_model_id` | str \| None | TEXT | Model ID string used for generation |
| `temperature` | float | REAL | LLM temperature for agent calls, default 0.7 |
| `user_instructions` | str | TEXT | User-provided additional instructions appended to system prompt, default `""` |
| `created_at` | datetime | TEXT (ISO) | Auto-set on create |
| `modified_at` | datetime | TEXT (ISO) | Auto-set on create/update |

### SQLModel Definition

```python
class ChatSession(SQLModel, table=True):
    __tablename__ = "chat_sessions"

    id: int = Field(primary_key=True)
    user_id: int = Field(foreign_key="users.id", index=True)
    world_id: int = Field(foreign_key="worlds.id", index=True)
    current_location_id: int | None = Field(default=None, foreign_key="world_locations.id")
    character_name: str
    character_description: str
    character_stats: str = "{}"
    world_stats: str = "{}"
    current_turn: int = Field(default=0)
    status: str = Field(default="active")
    llm_server_id: int | None = Field(default=None, foreign_key="llm_servers.id")
    llm_model_id: str | None = None
    temperature: float = Field(default=0.7)
    user_instructions: str = Field(default="")
    created_at: datetime
    modified_at: datetime
```

### Key Behaviors

- **Stats storage**: `character_stats` and `world_stats` are JSON strings mapping stat name → current value. Values follow `WorldStatDefinition` types (int as number, enum as string, set as JSON array). Parsed via `json.loads()`/`json.dumps()` in service layer.
- **Stat initialization**: On chat creation, iterate `WorldStatDefinition` rows for the world. For each, set `default_value` in the appropriate stats JSON (character-scoped or world-scoped).
- **Turn counter**: Starts at 0. Incremented after each complete user+assistant exchange.
- **Status**: Only `"active"` chats accept new messages. `"archived"` = read-only.

## 2. Chat Message Model

Individual messages in a chat session.

**File**: `backend/app/models/chat_message.py`

### Table: `chat_messages`

| Field | Type | SQLite Type | Notes |
|---|---|---|---|
| `id` | int64 (snowflake) | INTEGER | Primary key |
| `session_id` | int64 | INTEGER | FK → chat_sessions.id, indexed |
| `role` | str | TEXT | `"user"` / `"assistant"` / `"system"` |
| `content` | str | TEXT | Message text content |
| `turn_number` | int | INTEGER | Which turn this message belongs to, indexed |
| `tool_calls` | str \| None | TEXT | JSON array of tool call records, or None |
| `summary_id` | int64 \| None | INTEGER | FK → chat_summaries.id, nullable, indexed. Set when message is part of a summary. |
| `is_active_variant` | bool | INTEGER (0/1) | True = selected/active message |
| `created_at` | datetime | TEXT (ISO) | Auto-set on create |

### SQLModel Definition

```python
class ChatMessage(SQLModel, table=True):
    __tablename__ = "chat_messages"

    id: int = Field(primary_key=True)
    session_id: int = Field(foreign_key="chat_sessions.id", index=True)
    role: str
    content: str
    turn_number: int = Field(index=True)
    tool_calls: str | None = None
    summary_id: int | None = Field(default=None, foreign_key="chat_summaries.id", index=True)
    is_active_variant: bool = Field(default=True)
    created_at: datetime
```

### Tool Calls JSON Format

When an assistant message involved tool calls, `tool_calls` stores:

```json
[
  {
    "tool_name": "get_location_info",
    "arguments": {"location_id": "123456789"},
    "result": "{ ... location JSON ... }"
  }
]
```

Stored for debugging/display. Tool results are already incorporated into the assistant's `content`.

### Key Behaviors

- **Ordering**: By `turn_number` ASC, then `created_at` ASC within a turn.
- **summary_id**: FK to `chat_summaries.id`. Set when a summary covers this message. `NULL` for non-summarized messages. Check `summary_id IS NOT NULL` to determine if summarized. Denormalized back-reference for efficient querying (`WHERE summary_id = X`).
- **is_active_variant**: For regeneration. Active variant = True, superseded variants = False. Only active variants shown in chat view (except during variant selection).
- **System messages**: Used for the initial setup message at chat creation.

## 3. Regeneration Design

Regeneration uses the `is_active_variant` field on `ChatMessage` — no separate table needed.

### How It Works

1. User clicks "Regenerate" on the last assistant message.
2. Backend creates a **new** `ChatMessage` for the same `turn_number` with `is_active_variant=True`.
3. Previous assistant message for that turn → `is_active_variant=False`.
4. Multiple regenerations → multiple variant rows sharing `session_id` + `turn_number` + `role="assistant"`.
5. UI shows all variants for the latest turn, user can navigate between them.
6. When user sends the next message ("continue"), the selected variant keeps `is_active_variant=True`, all others are **deleted**.
7. Variants only exist for the **latest** turn.

### Variant Selection UI

- Shows "Variant 1/3" with prev/next arrows.
- Selecting a variant sets `is_active_variant=True` on chosen, `False` on others.
- "Continue" commits the selection and deletes non-active variants.

## 4. Chat State Snapshot Model

Records character stats and world stats at each turn boundary, enabling rewind.

**File**: `backend/app/models/chat_state_snapshot.py`

### Table: `chat_state_snapshots`

| Field | Type | SQLite Type | Notes |
|---|---|---|---|
| `id` | int64 (snowflake) | INTEGER | Primary key |
| `session_id` | int64 | INTEGER | FK → chat_sessions.id, indexed |
| `turn_number` | int | INTEGER | Turn this snapshot was taken after |
| `location_id` | int64 \| None | INTEGER | Character's location at this turn |
| `character_stats` | str | TEXT | JSON object — copy of session's character_stats |
| `world_stats` | str | TEXT | JSON object — copy of session's world_stats |
| `created_at` | datetime | TEXT (ISO) | Auto-set on create |

### SQLModel Definition

```python
class ChatStateSnapshot(SQLModel, table=True):
    __tablename__ = "chat_state_snapshots"

    id: int = Field(primary_key=True)
    session_id: int = Field(foreign_key="chat_sessions.id", index=True)
    turn_number: int
    location_id: int | None = None
    character_stats: str = "{}"
    world_stats: str = "{}"
    created_at: datetime
```

### Key Behaviors

- **Snapshot timing**: Taken after the assistant response for each turn.
- **Turn 0 snapshot**: Taken immediately after chat creation — initial stats and starting location.
- **Rewind**: Load snapshot for target turn, restore stats and location to session, delete snapshots with `turn_number > target`.
- **Composite index**: `(session_id, turn_number)` for efficient lookup.

## 5. Chat Summary Model

Stores compacted conversation sections.

**File**: `backend/app/models/chat_summary.py`

### Table: `chat_summaries`

| Field | Type | SQLite Type | Notes |
|---|---|---|---|
| `id` | int64 (snowflake) | INTEGER | Primary key |
| `session_id` | int64 | INTEGER | FK → chat_sessions.id, indexed |
| `start_message_id` | int64 | INTEGER | FK → chat_messages.id, first message in range (inclusive) |
| `end_message_id` | int64 | INTEGER | FK → chat_messages.id, last message in range (inclusive) |
| `start_turn` | int | INTEGER | Turn number of first message |
| `end_turn` | int | INTEGER | Turn number of last message |
| `content` | str | TEXT | Summary text produced by LLM |
| `created_at` | datetime | TEXT (ISO) | Auto-set on create |

### SQLModel Definition

```python
class ChatSummary(SQLModel, table=True):
    __tablename__ = "chat_summaries"

    id: int = Field(primary_key=True)
    session_id: int = Field(foreign_key="chat_sessions.id", index=True)
    start_message_id: int = Field(foreign_key="chat_messages.id")
    end_message_id: int = Field(foreign_key="chat_messages.id")
    start_turn: int
    end_turn: int
    content: str
    created_at: datetime
```

### Key Behaviors

- **Message range**: `start_message_id` through `end_message_id` inclusive. All messages in range get `summary_id` set to this summary's ID.
- **Sequential**: Summaries do not overlap. New summary starts from message after previous summary's `end_message_id`.
- **Context building order**: system prompt → [summary blocks ordered by start_turn ASC] → [raw non-summarized active messages].
- **Rewind interaction**: If rewinding to a turn within or before a summary range, the summary is deleted and `summary_id` set to `NULL` on affected messages.
